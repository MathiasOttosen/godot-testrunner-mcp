# Godot MCP Startup Resilience — Design Spec
*2026-04-10*

---

## Overview

Godot MCP currently assumes that a project can be launched with the default Godot runtime settings and that the project-side remote control autoload will become reachable quickly. In practice, startup can fail before project code runs, or the game can exit before MCP is usable.

This spec defines a resilient startup model for Godot MCP so it can:

- survive engine-level startup failures such as default log-file path crashes
- distinguish engine boot failure from project autoload failure
- avoid false "session did not connect" results when the game exited for a known reason
- support projects that intentionally run headless test autoloads
- expose actionable failure states instead of a single generic timeout

The motivating failure case is a Godot 4.6.1 project that crashed in `RotatedFileLogger::rotate_file()` during startup unless launched with an explicit `--log-file`, and whose headless path also auto-ran tests and exited before MCP could be used.

---

## Problem

### Observed failure modes

1. Godot crashes before project code runs.
   - Example: default `user://logs/...` setup crashes inside engine startup.
   - Result: MCP waits for a TCP connection that will never arrive.

2. Project autoloads terminate the process before MCP is ready.
   - Example: a headless-only `TestRunner` autoload executes, prints results, then calls `quit()`.
   - Result: MCP reports connection timeout even though the launch technically succeeded.

3. Remote-control scene navigation can fail during `_ready()`.
   - Example: changing scenes while the parent is still adding/removing children.
   - Result: startup error even though the TCP server is listening.

4. Project pausing can disable the remote-control process loop.
   - Example: `SceneTree.paused = true` also pauses the MCP autoload.
   - Result: `set_tree_paused(true)` works once, then the session becomes unresponsive.

### Why current behaviour is insufficient

- A timeout collapses distinct failures into one symptom.
- The user has no guidance on whether to fix the engine launch, the project autoloads, or the MCP addon.
- Headless and UI launch paths are treated as equivalent even when the project intentionally behaves differently in those modes.

---

## Goals

- Make startup failures diagnosable within one launch attempt.
- Recover automatically from common engine-level launch failures when recovery is safe.
- Separate launch mode concerns: UI inspection, runtime control, and test execution.
- Ensure the project-side remote control remains operable during pause and scene changes.
- Return structured failure reasons to the MCP client.

## Non-Goals

- Solving arbitrary Godot engine crashes beyond safe launch retries.
- Rewriting project-specific test runners.
- Hiding all project errors from the user.

---

## Design

## Feature 1: Multi-Stage Launch Pipeline

Godot MCP launch should become an explicit pipeline with checkpoints instead of a single "start process, wait for port" step.

### Stages

1. Preflight
   - Resolve project path
   - Choose launch mode: `ui`, `headless-mcp`, or `headless-tests`
   - Build command line

2. Spawn
   - Start Godot process
   - Capture stdout/stderr and process exit code

3. Early health window
   - Watch for:
     - immediate process exit
     - engine crash signatures
     - known log-path failures
     - MCP port open

4. Handshake
   - Connect to MCP port
   - Send a small command such as `ping` or `get_ui`
   - Confirm project-side remote control is responsive, not merely listening

5. Ready
   - Return a live session object with launch metadata

### Required client-visible states

`start_ui_session()` should return one of:

- `ready`
- `launch_failed_engine`
- `launch_failed_project`
- `launch_failed_autoload_exit`
- `launch_failed_timeout`
- `launch_recovered_with_fallback`

Each non-ready state should include:

- command used
- whether the process exited
- exit code if available
- last relevant stderr/stdout lines
- whether the MCP port ever opened
- whether a fallback launch was attempted

## Feature 2: Safe Launch Fallback for Log-File Failures

If startup fails before project code runs and the output matches a known log-path failure pattern, Godot MCP should retry automatically with an explicit safe log file path.

### Detection

Initial implementation should match either of:

- `Failed to open 'user://logs/`
- `RotatedFileLogger::rotate_file()`

### Fallback behaviour

Retry once with:

```text
--log-file <safe-temp-path>
```

Where `<safe-temp-path>` is generated by MCP, for example:

```text
/tmp/godot-mcp/<project_slug>/<timestamp>.log
```

### Constraints

- Only retry once.
- Only use this fallback for pre-project engine startup failures.
- Surface in the result that fallback was required.

### Rationale

This handles a class of engine/runtime environment failures without forcing per-project workarounds or requiring the user to edit launch commands manually.

---

## Feature 3: Launch Modes Must Be Explicit

Godot MCP should not treat all launches as the same operation.

### Modes

#### `ui`

Purpose: run the game for visual inspection and runtime interaction.

Default characteristics:
- non-headless
- `--mcp`
- optional scene override
- fallback `--log-file` allowed

#### `headless-mcp`

Purpose: run a headless runtime that is still expected to stay alive for MCP control.

Default characteristics:
- `--headless`
- `--mcp`
- optional scene override
- fallback `--log-file` allowed

#### `headless-tests`

Purpose: run automated tests and allow the process to exit normally.

Default characteristics:
- `--headless`
- no expectation that MCP will connect
- exit is not automatically treated as failure

### Why this matters

A project may intentionally attach test autoloads only in headless mode. Using `headless` for UI/session startup becomes incorrect for such projects. `start_ui_session()` should prefer `ui` by default unless the caller explicitly requests headless runtime control.

---

## Feature 4: Project-Side MCP Autoload Contract

Godot MCP should define a minimal contract that the project-side remote control script must satisfy.

### Required behaviours

1. Remote control must set:
   - `process_mode = Node.PROCESS_MODE_ALWAYS`

2. Scene changes requested during startup must be deferred.
   - Example: `call_deferred()` or equivalent wrapper around `change_scene_to_file()`

3. Remote control must support a minimal core command set:
   - `ping`
   - `get_ui`
   - `get_tree_paused`
   - `set_tree_paused`
   - `set_engine_time_scale`
   - `quit`

4. Command handlers must return structured `ok/error` payloads.

### Optional validation

Godot MCP should expose a compatibility check that validates the project-side addon version before session startup. If the project addon lacks required commands, the client should report `launch_failed_project` with a clear incompatibility message.

---

## Feature 5: Better Early-Exit Classification

If the Godot process exits before MCP handshake completes, Godot MCP should classify the exit using captured output.

### Classification heuristics

#### Engine startup failure

Indicators:
- crash backtrace before project logs
- `RotatedFileLogger`
- no project script output

#### Test-runner exit

Indicators:
- many `PASS:` / `FAIL:` lines
- explicit test summary
- normal or near-normal shutdown after suite execution

#### Project script failure

Indicators:
- `SCRIPT ERROR:`
- GDScript backtrace
- process exit after project output begins

### Returned metadata

Include:
- `classification`
- `summary`
- `evidence_lines`

This lets the client say "headless launch ran tests and exited" instead of "game did not connect within 20s".

---

## Feature 6: Session Handshake Should Verify Responsiveness

A listening TCP port is not sufficient. The process may be partially alive but not operational.

### Handshake protocol

After TCP connect:

1. Send `ping`
2. Wait for `{"ok": true, "pong": true}`
3. Optionally send `get_ui` or `get_tree_paused`
4. Only then mark the session ready

### Benefits

- Detects stale listeners or half-initialized sessions
- Confirms command loop is active
- Prevents "ready" from being reported when only the socket exists

## API Changes

## `start_ui_session`

Add optional parameters:

- `mode`: `"ui" | "headless-mcp" | "headless-tests"`
- `allow_logfile_fallback`: `bool = true`
- `log_file_path`: optional explicit override

Return shape should include:

```json
{
  "status": "ready | launch_failed_* | launch_recovered_with_fallback",
  "mode": "ui",
  "used_fallback": true,
  "fallback": {
    "type": "log_file_override",
    "path": "/tmp/godot-mcp/the_pattern/2026-04-10T..."
  },
  "diagnostics": {
    "classification": "engine_startup_failure",
    "evidence_lines": ["Failed to open 'user://logs/...'", "RotatedFileLogger::rotate_file()"]
  }
}
```

## `check_scaffold`

Extend it to report MCP protocol/version compatibility, not only file presence.

Example additions:

- required commands present
- remote control process mode status
- addon version hash

## Implementation Notes

### Client side

- Add launch output buffering for the first startup window.
- Add failure-pattern matching before timeout.
- Retry with `--log-file` when the pattern matches.
- Distinguish `ui` from `headless-mcp` in the launcher implementation.

### Project addon side

- Add `ping`
- Add pause/time-scale commands if absent
- Make startup scene changes deferred
- Ensure `PROCESS_MODE_ALWAYS`

### Backward compatibility

- If a project addon is old, MCP should still attempt startup.
- If handshake reveals missing commands, downgrade gracefully and report the missing capabilities.

## Rollout

### Phase 1

- Add launch failure classification
- Add `--log-file` fallback
- Add richer `start_ui_session()` result payload

### Phase 2

- Add protocol handshake with `ping`
- Add scaffold compatibility reporting

### Phase 3

- Split runtime launch modes formally in the public tool surface
- Add version negotiation between MCP client and project addon

## Acceptance Criteria

1. If Godot crashes on default `user://logs/...`, Godot MCP retries with `--log-file /tmp/...` and reports `launch_recovered_with_fallback` when successful.
2. If a headless test runner exits before MCP handshake, Godot MCP reports `launch_failed_autoload_exit` with test-runner classification instead of timeout.
3. If the remote control autoload pauses the tree, MCP commands still work.
4. If startup scene override is used, no `_ready()` child-removal error occurs.
5. `start_ui_session()` distinguishes "process never launched", "engine crashed", "project script failed", and "project exited intentionally".

---

## Risks

- Automatic retries can hide flaky environment issues if diagnostics are not surfaced clearly.
- Pattern-based failure detection may need maintenance across Godot versions.
- Some projects may rely on custom remote control addons that only partially match the contract.

---

## Open Questions

1. Should `--log-file` fallback be enabled globally or only after an engine crash signature match?
2. Should `start_ui_session()` default to `ui` mode for all projects, with headless requiring explicit opt-in?
3. Should Godot MCP own the project-side addon version, or support partial compatibility across versions?
